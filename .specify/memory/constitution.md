# Open-Meteo 気温予測アプリケーション憲法

## コア原則

### I. テスト駆動開発の徹底（絶対遵守）

すべての機能実装はテスト駆動開発（TDD）で進める。実装前にテストを作成し、仕様に対する検証を必須とする。

**必須要件:**

- テスト作成 → ユーザー承認 → テスト失敗確認 → 実装開始の順序を厳守
- Red-Green-Refactor サイクルの遵守
- 受入基準（Acceptance Criteria）は Given-When-Then 形式で記述
- 実装コード量に対するテストカバレッジ率80%以上を維持

**根拠:**
仕様と実装の乖離を最小化し、回帰バグを防止する。テストが仕様書として機能し、長期的なメンテナンス性を確保する。

**TDDサイクル:**

```mermaid
stateDiagram-v2
    [*] --> テスト作成
    テスト作成 --> ユーザー承認
    ユーザー承認 --> RED: テスト失敗確認
    RED --> GREEN: 最小実装
    GREEN --> REFACTOR: リファクタリング
    REFACTOR --> テスト実行
    テスト実行 --> テスト成功: パス
    テスト実行 --> GREEN: 失敗
    テスト成功 --> [*]
  
    note right of RED
        テストが失敗することを確認
        実装前の必須ステップ
    end note
  
    note right of GREEN
        テストを通す最小限の実装
        過剰な機能追加は禁止
    end note
  
    note right of REFACTOR
        コードの可読性・保守性向上
        テストはそのまま維持
    end note
```

### II. セキュリティ最優先

セキュリティ要件は機能要件より常に優先される。セキュリティ違反のコードは品質ゲートを通過できない。

**必須要件:**

- 機密データ（APIキー、個人情報、認証トークン等）の平文保存を禁止
- 暗号化・ハッシュ化の実装が必須
- 外部APIとの通信は HTTPS を強制
- XSS、CSRF、インジェクション攻撃への対策を実装前に設計
- セキュリティレビューを全PRで実施

**根拠:**
ユーザーデータ保護とアプリケーション信頼性の確保。セキュリティ問題は後から修正するより、設計段階で防ぐ方がコストが低い。

**セキュリティチェックフロー:**

```mermaid
flowchart LR
    A[コード変更] --> B{機密データ含む?}
    B -->|Yes| C[暗号化実装]
    B -->|No| D{外部通信あり?}
    C --> E[Linterチェック]
    D -->|Yes| F[HTTPS強制]
    D -->|No| E
    F --> E
    E --> G{脆弱性検出?}
    G -->|Yes| H[修正必須]
    G -->|No| I[セキュリティレビュー]
    H --> A
    I --> J{承認?}
    J -->|Yes| K[マージ可能]
    J -->|No| H
  
    style H fill:#ff6b9d,stroke:#c41e3a,color:#fff
    style K fill:#39ff14,stroke:#0f0,color:#000
```

### III. パフォーマンス定量化

パフォーマンス要件を定量的に定義し、受入基準に組み込む。測定不可能な要件は受け入れない。

**必須要件:**

- API レスポンスタイム: 95パーセンタイルで 200ms 以下
- 初回ページロード時間: 3秒以内（3G接続環境）
- グラフ描画処理: 100データポイントで 100ms 以下
- メモリ使用量: ブラウザで 100MB 以下
- パフォーマンステストを継続的インテグレーション（CI）に含める

**根拠:**
ユーザー体験の一貫性を保証。パフォーマンス劣化を早期検知し、リリース前に是正する。

**パフォーマンス監視マトリックス:**

| 指標          | 目標値           | 測定方法                 | 失敗時対応                     |
| ------------- | ---------------- | ------------------------ | ------------------------------ |
| APIレスポンス | 200ms以下（P95） | Performance API          | API最適化、キャッシュ導入      |
| 初回ロード    | 3秒以内（3G）    | Lighthouse               | バンドルサイズ削減、遅延ロード |
| グラフ描画    | 100ms以下        | performance.mark/measure | データ間引き、Canvas最適化     |
| メモリ使用量  | 100MB以下        | Chrome DevTools          | メモリリーク修正、GC最適化     |

**継続的監視フロー:**

```mermaid
sequenceDiagram
    participant Dev as 開発者
    participant CI as CI/CD
    participant Monitor as パフォーマンス監視
    participant Alert as アラート
  
    Dev->>CI: コミット＆プッシュ
    CI->>Monitor: パフォーマンステスト実行
    Monitor->>Monitor: 各指標を測定
  
    alt 基準値超過
        Monitor->>Alert: アラート送信
        Alert->>Dev: 修正要求
        Dev->>CI: パフォーマンス改善
    else 基準値内
        Monitor->>CI: テスト成功
        CI->>Dev: デプロイ可能
    end
```

### IV. 外部依存の再現性確保

外部依存パッケージはバージョンを固定し、環境の再現性を確保する。

**必須要件:**

- `package.json` / `package-lock.json` でバージョンを厳密に固定
- セマンティックバージョニング範囲指定（`^`, `~`）を避ける
- 依存パッケージの更新は専用ブランチで検証後にマージ
- 本番環境と開発環境のパッケージバージョン完全一致を保証

**根拠:**
「手元では動くが本番で動かない」問題を根絶。チーム全体で同一環境を保証し、デバッグ効率を向上させる。

**依存パッケージ管理ベストプラクティス:**

```json
// ❌ 悪い例（範囲指定）
{
  "dependencies": {
    "chart.js": "^4.0.0",    // 4.x.x の最新が入る（不確定）
    "express": "~5.0.0"      // 5.0.x の最新が入る（不確定）
  }
}

// ✅ 良い例（固定バージョン）
{
  "dependencies": {
    "chart.js": "4.4.0",     // 常に 4.4.0（確定）
    "express": "5.0.1"       // 常に 5.0.1（確定）
  }
}
```

### V. 仕様・実装乖離の検知と是正

仕様ドキュメントと実装コードの乖離をレビューで検知し、即座に是正する。

**必須要件:**

- 仕様変更は必ず `specs/` ディレクトリのドキュメントに反映
- コードレビュー時に仕様ドキュメントとの整合性を確認
- 実装が仕様を超えた機能追加を禁止（スコープクリープ防止）
- 仕様変更履歴を Git コミットログと紐付け

**根拠:**
ドキュメントと実装の不一致は将来の保守コストを指数関数的に増加させる。常に同期を維持することで技術的負債を抑制する。

**仕様・実装同期フロー:**

```mermaid
flowchart LR
    subgraph "仕様変更時"
        A[仕様変更要求] --> B[spec.md更新]
        B --> C[plan.md更新]
        C --> D[tasks.md更新]
        D --> E[仕様レビュー]
    end
  
    subgraph "実装時"
        E --> F[実装開始]
        F --> G[コードレビュー]
        G --> H{仕様と一致?}
        H -->|No| I[仕様再確認]
        I --> J{仕様が正?}
        J -->|Yes| K[実装修正]
        J -->|No| L[仕様修正]
        L --> B
        K --> G
        H -->|Yes| M[マージ]
    end
  
    style I fill:#ff6b9d,stroke:#c41e3a,color:#fff

```

**レビューチェックリスト（実装ブランチ）:**

- [ ] 仕様に記載されていない機能を追加していない
- [ ] テストケースが受入基準（Acceptance Criteria）をカバーしている
- [ ] APIレスポンス形式が契約（contracts/）と一致している
- [ ] パフォーマンス目標を達成している
- [ ] セキュリティ要件を満たしている

## セキュリティ・品質制約

### セキュリティ制約

1. **機密データ保護:**

   - API キーは環境変数または暗号化ストレージに保存
   - コードベースに機密情報を含めない（.gitignore で除外）
   - ログ出力に機密情報を含めない
2. **依存パッケージ監査:**

   - `npm audit` を CI パイプラインで実行
   - 高リスク脆弱性が検出された場合はビルド失敗
   - 月次で依存パッケージセキュリティレビューを実施
3. **入力検証:**

   - ユーザー入力は必ずバリデーション
   - HTML エスケープ処理を施す
   - API レスポンスの型チェックを実装

**セキュリティ脅威モデル（STRIDE）:**

| 脅威                                         | 対策                             | 実装場所        |
| -------------------------------------------- | -------------------------------- | --------------- |
| **S**poofing（なりすまし）             | HTTPS通信、CORS設定              | API通信レイヤー |
| **T**ampering（改ざん）                | 入力検証、CSP設定                | フロントエンド  |
| **R**epudiation（否認）                | アクセスログ記録                 | サーバーサイド  |
| **I**nformation Disclosure（情報漏洩） | 機密データ暗号化、ログマスキング | 全レイヤー      |
| **D**enial of Service（DoS）           | レート制限、タイムアウト設定     | API通信レイヤー |
| **E**levation of Privilege（権限昇格） | 最小権限原則、認可チェック       | 認証レイヤー    |

**セキュリティチェックリスト（全PR必須）:**

- [ ] 機密情報がコードに含まれていない
- [ ] ユーザー入力に対するバリデーション実装済み
- [ ] XSS対策（HTMLエスケープ）実装済み
- [ ] HTTPS通信を強制している
- [ ] `npm audit` で高リスク脆弱性なし
- [ ] ログに機密情報が含まれていない
- [ ] Content Security Policy (CSP) 設定済み（該当する場合）

### 品質制約

1. **コード品質基準:**

   - Linter（ESLint 等）エラーゼロを維持
   - コードフォーマッター（Prettier 等）で統一
   - 複雑度（Cyclomatic Complexity）15 以下
   - 関数は単一責任原則に従う（50行以内推奨）
2. **ドキュメント品質:**

   - 全ての公開関数に JSDoc コメント
   - README に環境構築手順を記載
   - Mermaid 図（v11準拠）でフロー・アーキテクチャを可視化
3. **ユーザー体験一貫性:**

   - デザインシステム（色、フォント、スペーシング）を定義
   - アクセシビリティ（ARIA ラベル、キーボード操作）を考慮
   - レスポンシブデザイン（モバイル・タブレット・デスクトップ）

**コード品質メトリクス:**

| メトリクス       | 基準値          | 測定ツール          | 違反時対応           |
| ---------------- | --------------- | ------------------- | -------------------- |
| Lintエラー       | 0               | ESLint              | マージブロック       |
| 複雑度           | ≤15            | ESLint (complexity) | リファクタリング必須 |
| テストカバレッジ | ≥80%           | Jest                | 追加テスト作成       |
| 関数行数         | ≤50行          | 手動レビュー        | 関数分割             |
| ドキュメント率   | 100%（公開API） | JSDoc検証           | コメント追加         |

#### 仕様ブランチ（main ブランチから派生）

仕様変更・新機能仕様は専用ブランチで作成する。

```bash
git checkout main
git checkout -b spec/<番号>-<短い名前>
# 例: spec/001-temperature-chart
```

**対象ファイル:**

- `.specify/specs/<番号>-<機能名>/spec.md` - ユーザーストーリー、要件定義
- `.specify/specs/<番号>-<機能名>/plan.md` - 実装計画、技術スタック
- `.specify/specs/<番号>-<機能名>/tasks.md` - タスク分解、依存関係
- `.specify/specs/<番号>-<機能名>/research.md` - 技術調査結果（必要時）
- `.specify/specs/<番号>-<機能名>/data-model.md` - データモデル定義（必要時）
- `.specify/specs/<番号>-<機能名>/contracts/` - API契約定義（必要時）

#### 実装ブランチ（仕様ブランチから派生）

実装は仕様ブランチから分岐する。

```bash
git checkout spec/001-temperature-chart
git checkout -b feature/impl-001-temperature-chart
```

**対象ファイル:**

- `src/`, `tests/` 配下の実装ファイル
- `package.json`, `package-lock.json` の依存追加
- 実装に関連するドキュメント（必要時）

### マージ基準

- [ ] パフォーマンステストが基準を満たす
- [ ] セキュリティレビュー完了
- [ ] 仕様ドキュメントとの整合性確認
- [ ] レビュアー承認（最低1名）

### レビュープロセス

1. **セルフレビュー:**

   - 憲法チェックリストを確認
   - テスト実行・カバレッジ確認
   - Linter 実行
2. **ピアレビュー:**

   - 仕様との整合性
   - セキュリティ脆弱性チェック
   - パフォーマンス影響評価
   - コード可読性・保守性
3. **承認基準:**

   - 重大変更（API 変更、データモデル変更）は2名以上の承認必須
   - 軽微な変更（ドキュメント修正、タイポ修正）は1名承認で可

### 同時開発の調整

フロントエンドとバックエンドを同時開発する場合:

1. 契約（API 仕様）を先に定義
2. モックサーバーで並行開発
3. 統合テストで最終検証

### 遵守検証

- すべてのプルリクエストで憲法遵守を確認
- レビューチェックリストに憲法項目を含める
- 四半期ごとに憲法適合性監査を実施

**プルリクエストチェックリスト（必須）:**

```markdown
## 憲法遵守確認

### I. テスト駆動開発
- [ ] テストを実装前に作成した
- [ ] テストが失敗することを確認した
- [ ] 最小限の実装でテストを通した
- [ ] テストカバレッジ80%以上

### II. セキュリティ最優先
- [ ] 機密データの平文保存なし
- [ ] HTTPS通信を使用
- [ ] XSS/CSRF対策実装済み
- [ ] `npm audit` でクリティカル脆弱性なし

### III. パフォーマンス定量化
- [ ] APIレスポンスタイム200ms以下
- [ ] 初回ページロード3秒以内
- [ ] グラフ描画100ms以下
- [ ] メモリ使用量100MB以下

### IV. 外部依存の再現性確保
- [ ] `package.json` でバージョン固定
- [ ] `package-lock.json` を更新
- [ ] セマンティックバージョニング範囲指定なし

### V. 仕様・実装乖離の検知と是正
- [ ] 仕様ドキュメントと実装が一致
- [ ] スコープクリープなし
- [ ] 仕様変更をドキュメント反映

### レビュアー承認
- [ ] レビュアー: [NAME] - 承認日: [YYYY-MM-DD]
```
